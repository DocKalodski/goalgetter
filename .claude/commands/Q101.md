| ================================================== |

| **Chapter 1: THE VISION**                          |
| *"Every great application starts with an idea"*    |

>

**THE STORY**

Imagine you have an idea for a new application. Maybe it's a sales dashboard,\
a customer portal, or an API service. Traditionally, you'd:
- Write requirements documents manually
- Design the architecture yourself
- Code everything line by line
- Hope you didn't miss anything

With Q101, you start differently. You start with **ideation and research**...

---

**IDEATION & RESEARCH COMMANDS**

| Command | Purpose |
|---------|---------|
| **/ideate** | Guided brainstorming with Socratic questioning |
| | - 7 methodology modes (5 Whys, G.R.O.W., SCAMPER, etc.) |

| | **Output:** `idea-{topic}-{id}.md` |

>

|---------|---------|

**REQUIREMENTS COMMANDS**

| Command | Purpose |
|---------|---------|
| **/initialize** | Understand what you want to build |

| | - Creates requirements context for next step |

| ================================================== |

>

**THE STORY**

Your application is built, but before it goes live, it needs to pass quality gates and security checks...

---

**THE QUALITY PIPELINE**

| Command | Purpose |
|---------|---------|
| **/prepare** | Sets up the environment |
| | - Installs dependencies (pip, npm) |
| | - Configures .env from templates |
| | - Validates prerequisites |

>

| Command | Purpose |
|---------|---------|
| **/evaluate** | Runs comprehensive quality checks |
| | - Health checks - Does the app start? |
| | - Test suites - Do all tests pass? |
| | - Coverage - How much code is tested? |
| | - Quality metrics - Linting, type checking |
| | **Output:** `EVALUATION-REPORT.md` |

>

| Command | Purpose |
|---------|---------|
| **/iterate** | Fixes issues found in evaluation |
| | - Fix mode: Address EVALUATION-REPORT.md issues |
| | - Feature mode: Add new capabilities |
| | - Refactor mode: Improve code structure |
| | **Usage:** |
| | `/iterate` - Fix evaluation issues |
| | `/iterate "Add feature X"` - Add specific feature |
| | `/iterate --refactor` - Smart refactoring |

>

| Command | Purpose |
|---------|---------|
| **/secure** | Security assessment (REQUIRED for production) |
| | - OWASP Top 10 vulnerability scan |
| | - Authentication/authorization review |
| | - Secrets management check |
| | - Input validation verification |
| | - Applies security fixes with approval |
| | **Output:** `SECURITY-REPORT.md` |

---

**THE ITERATE LOOP**

/evaluate and /iterate work together in a continuous loop:

| Step | Outcome | Next Action |
|------|---------|-------------|
| /evaluate | Issues Found | /iterate (fix issues) |
| /iterate | Fixes Applied | /evaluate (verify) |
| /evaluate | All Pass | /secure (lock down) |

---

**KEY INSIGHT**

Quality isn't a one-time check - it's a loop. You iterate until the application meets your standards, then lock it down with security hardening.

---

**DUAL DEVELOPMENT PARADIGM**

Q101 offers two development modes:

| Mode | Command | Best For |
|------|---------|----------|
| **Step-by-Step** | /execute | Review-intensive projects |
| | | - Design checkpoint before coding |
| | | - Human approval at each phase |
| | | - Good for learning/demos |
| **Autonomous** | /autonomous | Large-scale implementations |
| | | - Feature-by-feature execution |
| | | - Automatic checkpointing |
| | | - Minimal human intervention |

---

**TWO-AGENT ARCHITECTURE**

The /autonomous command uses two specialized agents:

| Agent | Role |
|-------|------|
| @autonomous_initializer | Session 1: Create feature list |
| | - Detect/confirm tech stack |
| | - Generate 20-200+ features |
| | - Establish session state |
| @autonomous_coder | Sessions 2+: Implement features |
| | - One feature per session |
| | - Verify no regressions |
| | - Git checkpoint after each |

---

**SESSION MANAGEMENT**

| Command | Purpose |
|---------|---------|
| `/autonomous` | Start new session (20 session limit) |
| `/autonomous --full` | Unlimited sessions (run to completion) |
| `/autonomous --status` | Check current progress |
| `/autonomous --resume` | Continue from checkpoint |
| `/autonomous --pause` | Save state and exit gracefully |
| `/autonomous --features` | View feature list |

---

**EXTERNAL STATE PERSISTENCE**

Autonomous mode uses external files for state:

| File | Purpose |
|------|---------|
| `feature-list.json` | All features with status |
| `session-state.json` | Current session context |
| `progress.txt` | Human-readable progress |
| Git commits | Checkpoint after each feature |

---

**FEATURE COMPLEXITY TIERS**

| Tier | Steps | Example |
|------|-------|---------|
| Simple | 2-5 | Add button, create component |
| Medium | 5-10 | Implement form validation |
| Complex | 10+ | Build authentication system |

---

**KEY INSIGHT**

Autonomous mode is for when you trust the agents to run. Start a session,\
go get coffee, come back to find features implemented. The checkpointing\
means you never lose progress, and resume picks up exactly where you left off.
